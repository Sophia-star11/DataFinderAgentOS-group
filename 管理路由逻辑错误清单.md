## 一、瞭望采集（`app/controllers/admin_watch.py`）

### 1.1 source_type 存而不用 — 所有源都当百度新闻解析
- **文件**: `admin_watch.py:111`
- **错误**: `_fetch_from_source()` 无论 `source_type` 是什么，都硬编码调用 `_parse_baidu_news()`
- **影响**: RSS、JSON API 类型的瞭源采集时解析失败或返回空
- **应然**: 根据 `source["source_type"]` 分发到不同的解析器

### 1.2 HTTP method 被忽略
- **文件**: `admin_watch.py:94`
- **错误**: `urllib.request.Request(url, method="GET")` 硬编码为 GET
- **影响**: 配置了 POST 方法的瞭源实际仍以 GET 请求
- **应然**: 使用 `source["method"]`

### 1.3 URL 拼接缺失分隔符处理
- **文件**: `admin_watch.py:85`
- **错误**: `url_template + urllib.parse.urlencode(params)` — 如果模板末尾没有 `?` 或 `&`，拼接出的 URL 非法
- **影响**: 新增瞭源时若 url_template 末尾缺少分隔符，采集请求失败
- **应然**: 检测并自动添加 `?` 或 `&`

### 1.4 source_ids 解析无异常保护
- **文件**: `admin_watch.py:43`
- **错误**: `[int(x) for x in source_ids.split(",") if x.strip()]` — 如果 `source_ids` 包含非数字字符，直接抛出 `ValueError`
- **影响**: 前端传参异常时会导致 500
- **应然**: 用 try-except 包裹或使用 `str.isdigit()` 过滤

## 二、模型引擎（`app/controllers/admin_model.py`）

### 2.1 ModelUpdateApiHandler 遗漏 temperature 字段
- **文件**: `admin_model.py:87-88`
- **错误**: 更新循环的 key tuple 中缺少 `"temperature"`：
  ```python
  for key in ("name", "provider", "model_name", "api_base_url", "api_key",
               "max_tokens", "token_count", "status", "category",
               "system_prompt", "top_p", "context_length"):
  ```
  `"temperature"` 不在其中。虽然类型转换分支写了 `temperature`，但迭代没包含它，所以永远无法更新
- **影响**: 用户编辑模型时无法修改 temperature 参数
- **应然**: 在 tuple 中加入 `"temperature"`

### 2.2 同样遗漏 sort_order
- **文件**: `admin_model.py:87`
- **错误**: `"sort_order"` 不在 key tuple 中
- **影响**: 无法调整模型的排序顺序

### 2.3 SSE 流式 token 计数不准确
- **文件**: `admin_model.py:234`
- **错误**: `token_count += 1` — SSE 的每个 `delta.content` 块是一个 token 片段（可能是子词），不是完整 token
- **影响**: token 使用量统计严重偏高
- **应然**: 用 `usage.completion_tokens` 做精确统计，或做更合理的估算

## 三、功能管理（`app/controllers/admin_function.py`）

### 3.1 FunctionUpdateApiHandler 全字段覆盖
- **文件**: `admin_function.py:76-84`
- **错误**: 所有字段（包括空字符串）都打包到 updates 中发送：
  ```python
  updates = {
      "name": name, "code": code, "icon": icon, "route": route,
      "sort_order": sort_order, "parent_id": parent_id, "status": status
  }
  ```
  前端编辑时如果某个字段不填（如 route），会用空字符串覆盖数据库原有值
- **影响**: 部分更新导致字段被意外清空
- **应然**: 只传非空字段，或前端保证必填

## 四、用户管理（`app/controllers/admin_user.py`）

### 4.1 UserUpdateApiHandler 参数未判空
- **文件**: `admin_user.py:73-80`
- **错误**: `get_argument("user_id", "")` 后直接 `int(user_id)`，若为空字符串抛出 `ValueError`
- **影响**: 攻击者或前端异常可导致 500
- **应然**: 先判空再转换

### 4.2 UserDeleteApiHandler 参数未判空
- **文件**: `admin_user.py:129-132`
- **错误**: 同上，`user_id` 未判空就 `int()` 转换
- **影响**: 同 4.1

### 4.3 UserCreateApiHandler 默认 role_id 可能不存在
- **文件**: `admin_user.py:56`
- **错误**: `role_id = int(self.get_argument("role_id", 2))` — 硬编码默认值 2，如果数据库中没有 role_id=2 的记录，用户创建成功但关联到不存在的角色
- **影响**: 用户可能变成"无角色"用户

## 七、认证模块（`app/controllers/auth.py`）

### 7.1 AdminLoginHandler 未检查已登录状态
- **文件**: `auth.py:64`
- **错误**: `LoginHandler.get()` 有 `if self.current_user: self.redirect("/index")`，但 `AdminLoginHandler.get()` 没有 — 已登录的管理员仍然看到登录页
- **影响**: 用户体验不一致
- **应然**: 同样检查 `self.current_user`，已登录则跳转 `/admin/index`

### 7.2 Login/AdminLogin POST 参数无默认值
- **文件**: `auth.py:15-16, 68-69`
- **错误**: `self.get_body_argument("username")` 没有默认值 — 若请求缺少该字段，Tornado 直接抛 `MissingArgumentError(400)`，自定义错误提示永远不会触发
- **影响**: 用户看到的不是友好的"用户名或密码不能为空"而是 Tornado 默认错误页
- **应然**: 改用 `self.get_body_argument("username", "")` 并做空值检查

### 7.3 Logout/AdminLogout 为 POST-only
- **文件**: `auth.py:29-31, 82-85`
- **错误**: 退出路由定义为 POST，但前端通常用 `<a>` 链接或 GET 请求触发退出
- **影响**: 如果前端未使用表单 POST，点击退出按钮会 405
- **应然**: 检查前端模板的退出按钮实现方式，或将 handler 改为 `get`+`post` 双支持

### 7.4 FaceLoginHandler 无实际人脸验证
- **文件**: `auth.py:51-61`
- **错误**: "人脸登录"仅检查用户名存在且状态为 1，无任何生物特征验证
- **影响**: 安全性等同于直接输入密码登录，存在安全风险
- **应然**: 移除或对接真实人脸识别 API

## 十、数据仓库与深度采集（`app/controllers/admin_warehouse.py`）

### 10.1 深度采集任务重启丢失
- **文件**: `admin_warehouse.py:96-121`
- **错误**: `IOLoop.current().add_callback(...)` 启动后台协程，没有任务队列或持久化调度
- **影响**: Tornado 服务器重启后，所有 running/pending 的任务丢失且不可恢复

### 10.2 _find_deep_collect_employee 中文耦合
- **文件**: `admin_warehouse.py:88`
- **错误**: `"采集专员" in emp.get("name", "")` — 硬编码中文名称匹配
- **影响**: 如果用户的数字员工改名，即使类型匹配也无法自动识别

### 10.3 crawl4ai 懒导入
- **文件**: `admin_warehouse.py:129`
- **错误**: `from crawl4ai import AsyncWebCrawler` 在函数内部导入
- **影响**: 如果 `crawl4ai` 未安装，错误在运行时才暴露，而不是启动时

### 10.4 DeepCollectResultApiHandler 回退逻辑过于复杂
- **文件**: `admin_warehouse.py:516-552`
- **错误**: 多层嵌套的 fallback 逻辑（deep_collect_data → task.result_data → 回退查询），难以维护
- **影响**: 维护成本高，容易引入新 bug

## 十二、控制台 / 大屏（`app/controllers/home.py`）

### 12.1 DashboardStatsApiHandler / AdminIndexHandler 统计逻辑重复
- **文件**: `home.py:19-73, 82-105`
- **错误**: 两个 handler 有几乎完全相同的 SQL 查询逻辑
- **影响**: 修改一个忘记改另一个会导致数据不一致，违反 DRY 原则

### 12.2 OpinionScanApiHandler 全表扫描无限制
- **文件**: `home.py:408-466`

- **影响**: 数据量大时严重性能问题，且无防重复触发机制（前端随意频繁调用）

### 12.3 SensitiveWordsApiHandler 无分页
- **文件**: `home.py:480`
- **错误**: `conn.execute("SELECT * FROM sensitive_words ORDER BY id").fetchall()` — 返回全部敏感词
- **影响**: 词库量大时前端加载缓慢

### 13.1 get_user_info 每次渲染都查库
- **文件**: `base.py:57-60`
